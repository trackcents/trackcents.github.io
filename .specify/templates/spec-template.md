### Privacy Posture (Principle I)
- What user data does this feature read, write, or transmit?
- Where does that data live during the feature's operation (device only / encrypted blob in cloud / etc.)?
- Confirm: no plaintext bank statement content or parsed transactions leaves the device.

### Accuracy Posture (Principle II)
- Does this feature touch monetary values, totals, or imports? If yes:
  - Confirm integer minor units are used throughout.
  - Confirm checksum invariant is applied where statements are imported.
  - List the failure modes that must be loud (refused, flagged) rather than silent.

### Data Provenance (Principle VI)
- If this feature creates or modifies transaction-related data, confirm the full provenance fields are populated: `bank_name`, `account_type`, `account_nickname`, `account_last_4`, `statement_id`, `raw_text`, `parser_name`, `parser_version`, `pdf_source_hash`.
- If this feature filters / displays / aggregates transactions, confirm it does NOT branch on specific bank names in the logic.

### AI Use (Principle IX)
- Does this feature use AI? If yes:
  - Confirm it is the client-side categorization classifier only.
  - Confirm the user can disable it without losing core functionality.
  - Confirm the model sees only parsed transaction descriptions, never raw PDFs.
- If the feature proposes any other AI use, STOP — this requires a constitutional amendment.

### Cost & Hosting (Principle III)
- Confirm this feature does not introduce a recurring monthly cost beyond the $0/$20 ceiling.
- Confirm this feature does not require us to run a server that touches user data.
