=== IV&V REPORT — card-payment (CC-payment descriptor recognizer) (2026-05-26) ===

Module: src/lib/util/card-payment.ts (`detectCardPayment`, `bankNameMatchesIssuer`,
`ISSUER_BANK_NAMES`). Tier: Core, money-path-adjacent — it feeds the reconciliation
matcher (US-P1-D); a false positive can drive a WRONG reconciliation link. The
reconciliation IV&V had treated this as a trusted oracle; this run verifies it and
closes the last gap in the US-P1-D pipeline (detect → match → attribute → drill).

**Lean staged run (regex/pattern table = low mutation signal): Stage 1 only, ~4 min.**

Independence: VE read only the contract + spec (US-P1-D) + research §R12 + the ivv
config; never card-payment.ts or the Designer's tests. No mirror regex model (would
share blind spots) — directed hand-derived goldens + adversarial probes were the oracle.

RESULT: **DISCREPANCY FOUND → fixed (D25).** detectCardPayment's recognition + the
no-false-positive safety property were clean; the bug was in the issuer↔bank matcher.

  • **D25 (CONFIRMED BUG, fixed):** `bankNameMatchesIssuer` used a loose case-insensitive
    SUBSTRING match, so a bank whose name merely contains a short alias matched the wrong
    issuer: `"Citizens Bank"`→Citi, `"Pineapple Credit Union"`→Apple, `"purchase…"`→Chase
    (chase ⊂ purchase), `"BoARD…"`→BofA — 8 independent false positives. Contract INV3
    mandates `false`; a wrong match mis-routes a reconciliation link (the exact US-P1-D
    hazard). **Fix:** EXACT case-insensitive match against the alias list. Decisive
    evidence it's safe + sufficient: every CC adapter sets `bank_name` to a plain
    canonical alias (`'Robinhood'`, `'Bank of America'`, `'Chase'`, `'Discover'`,
    `'American Express'`) — all exact aliases — so exact-match covers production with ZERO
    false positives and fails CLOSED (a future suffixed form yields a missing link, which
    is safe + fixable, never a wrong link). An obsolete designer test that asserted the
    removed substring behavior on a non-adapter input (`"Robinhood Credit, Inc."`) was
    updated to the corrected semantics + a no-false-positive regression test. pnpm verify
    GREEN (1336).

Checked (Stage 1, all pass post-fix): per-issuer recognition goldens (Chase/Discover
last-4 + padding; no-last-4 issuers → null); INV1 no-false-positive (12 issuer-name-in-
non-payment descriptors → null); INV2 last-4 fidelity (Amex M-code not read as last-4);
INV3 alias soundness (8 substring-collision banks → false after fix); INV4 never-throws;
INV5 alias-table covers every issuer.

Tests promoted: tests/unit/independent/card-payment.ivv.test.ts (57 tests). Mutation
deferred on purpose (regex table, low signal); bench-has-teeth proven directly — the
INV3 probes failed against the buggy DUT and pass against the fix.

Honest residual: no mirror model for detectCardPayment (directed goldens cover the
DOCUMENTED descriptor forms only; false negatives on undocumented formats are out of
scope + safe per the recognizer design); no mutation score (deferred); INV1 is 12
adversarial cases, not exhaustive over all issuer-name-bearing strings. Scope note: the
exact-match fix assumes bank_name is adapter-controlled (true today — constitution VI
layering); if bank_name ever becomes free user text, revisit.

Pipeline status: with this run, the ENTIRE US-P1-D drill-down chain is independently
